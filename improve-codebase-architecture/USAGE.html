The architecture-review cluster

Four skills that work as one workflow: scan a codebase for shallow modules, render the candidates as a visual report, grill through the chosen one, and keep the domain model current as decisions crystallise.

Entry point

improve-codebase-architecture

Run this when you want to find and fix architectural friction.

Helpers

  • codebase-design — vocabulary + principles
  • domain-modeling — glossary + ADR maintenance
  • grill-me — interactive design interview

When to use it

Trigger improve-codebase-architecture when any of these are true:

  • You suspect modules are shallow — interface nearly as complex as the implementation.
  • You have to bounce between many small files to understand one concept.
  • Tests exist, but the bugs they don't catch live in the call sites, not the units.
  • Two modules that should be independent keep leaking across their seam.
  • You want a structured "what should we deepen first?" answer, not a vibes-based one.

Don't trigger it when the real problem is performance, a missing feature, or a concrete bug — those are different skills.

The three phases

The skill runs in three phases. The agent drives the first two, you drive the third.

Phase 1

Explore

The agent walks the codebase looking for architectural friction. It:

  1. Reads CONTEXT.md (the domain glossary) and any relevant ADRs in docs/adr/.
  2. Spawns an Explore sub-agent to walk the code organically — no rigid heuristic checklist.
  3. Notes where it experiences friction: shallow modules, leak-prone seams, low-locality code, untested surfaces.
  4. Applies the deletion test to each suspect: would deleting it concentrate complexity, or just move it?

Output: a mental shortlist of deepening candidates.

Phase 2

Present as an HTML report

The agent writes a self-contained HTML report to the OS temp dir and opens it for you. Path pattern:

<tmpdir>/architecture-review-<timestamp>.html

Where <tmpdir> is $TMPDIR, then /tmp, then %TEMP% on Windows. Nothing lands in the repo.

The report is built with Tailwind via CDN and Mermaid via CDN, mixed with hand-crafted SVG/divs. Each candidate gets a card with:

  • Files — which modules are involved.
  • Problem — one sentence on what hurts.
  • Solution — one sentence on what changes.
  • Benefits — in terms of locality and leverage.
  • Before / After diagram — custom-drawn, side by side.
  • Recommendation strength — Strong / Worth exploring / Speculative.

A Top recommendation section picks one candidate to tackle first and links to its card.

After the file is written, the agent asks: "Which of these would you like to explore?"

Phase 3

Grilling loop

Once you pick a candidate, the agent hands off to grill-me:

  1. One question at a time, with a recommended answer. No multi-question bombs.
  2. Walk down the design tree: constraints → dependencies → shape of the deepened module → what sits behind the seam → what tests survive.
  3. Side effects happen inline as decisions crystallise (see below).
  4. If the user rejects a candidate with a load-bearing reason, the agent offers to record it as an ADR.

After the grilling, the agent may also offer to spawn parallel sub-agents to design the interface twice (or three, or four times) — see codebase-design/DESIGN-IT-TWICE.md.

Side effects on your repo

The cluster can write to two places in the project. Both are lazy — files appear only when there's something to record.

CONTEXT.md

Domain glossary

Created at the repo root (or under a bounded context, see CONTEXT-MAP.md). Updated inline when terms crystallise. Format: opinionated terms with _Avoid_ lists.

docs/adr/

Architecture decision records

Sequentially numbered (0001-slug.md). An ADR is only offered when all three are true: hard to reverse, surprising without context, real trade-off.

ADR conflicts. If a candidate contradicts an existing ADR, the agent only surfaces it when the friction is real. A theoretical refactor an ADR forbids is not a candidate.

The vocabulary

Spoken by the agent throughout. Use these terms in your replies so the agent doesn't drift into "component," "service," "API," or "boundary."

Module
Anything with an interface and an implementation. Scale-agnostic: a function, a class, a package, a tier-spanning slice. Avoid: unit, component, service.
Interface
Everything a caller must know: signatures, invariants, ordering, error modes, config, performance. Avoid: API, signature.
Implementation
What's inside the module. Distinct from adapter: an adapter is the role it plays at a seam.
Depth
Leverage at the interface. A deep module hides lots of behaviour behind a small interface. A shallow one has an interface nearly as wide as its implementation.
Seam
A place where you can alter behaviour without editing in that place — where the interface lives. Avoid: boundary (overloaded with DDD).
Adapter
A concrete thing that satisfies an interface at a seam. Role, not substance.
Leverage
What callers get from depth: more capability per unit of interface learned.
Locality
What maintainers get from depth: change, bugs, knowledge, and verification concentrate in one place.

Principles

  • The deletion test. Imagine deleting the module. If complexity vanishes, it was a pass-through. If complexity reappears across N callers, it was earning its keep.
  • The interface is the test surface. Callers and tests cross the same seam. If you want to test past the interface, the module is the wrong shape.
  • One adapter means a hypothetical seam. Two means a real one. Don't introduce a port unless at least two adapters are justified (typically production + test).

Deep vs shallow

The report visualises this with two rectangles per module — interface surface area vs implementation. The deepening pushes the seam up:

Shallow (avoid)

Large interface — many methods, complex params
Thin implementation — just passes through

Interface leaks. Bugs hide in callers. Tests layer on top of each other.

Deep (aim for)

Small interface — few methods, simple params
Deep implementation — complex logic hidden

Bugs concentrate. Tests hit one surface. Callers stay thin.

Using the helpers standalone

Each helper works on its own. You don't have to run the full cluster to use them.

codebase-design

When the user wants to design or improve a module's interface, find deepening opportunities, decide where a seam goes, or make code more testable / AI-navigable. Reference docs: DEEPENING.md (dependency categories, testing strategy) and DESIGN-IT-TWICE.md (parallel sub-agent pattern for alternative interfaces).

domain-modeling

When the user wants to pin down domain terminology, record an architectural decision, or maintain the domain model inline. Reference docs: CONTEXT-FORMAT.md (glossary layout) and ADR-FORMAT.md (decision record format).

grill-me

Anytime you want to be interviewed relentlessly about a plan or design — outside the architecture review flow as well as inside it. One question at a time, with a recommended answer per question.

When the agent offers an ADR

An ADR is only offered when all three are true:

1. Hard to reverse. Changing your mind later has a meaningful cost.

2. Surprising without context. A future reader will look at the code and wonder why.

3. Real trade-off. Genuine alternatives existed and one was picked for specific reasons.

Ephemeral reasons ("not worth it right now") and self-evident choices don't qualify. The bar is "would a future explorer need this to avoid re-suggesting the same thing?"
